Resumen del Proyecto

Easy Kontrol es una plataforma SaaS multi-rol para la gestión integral de proveedores y personal contratista. Permite a empresas administradoras registrar, aprobar, rechazar y hacer seguimiento al personal de sus empresas contratistas (proveedores), incluyendo gestión documental, control de vencimientos, evaluación de proveedores y auditoría completa de acciones.

Rol Admin
Ve y gestiona todo el sistema: aprueba personal, crea usuarios, evalúa proveedores, accede a auditoría y reportes.
Rol Proveedor
Registra su propio personal, sube documentos, consulta estado de aprobaciones y visualiza vencimientos de su empresa.
Acceso Público
Formulario de registro sin cuenta para que el personal se registre directamente sin que el proveedor lo haga.

Arquitectura de Base de Datos

El proyecto usa PostgreSQL vía Supabase con RLS activado en todas las tablas. La base de datos tiene 12 tablas, 3 triggers, 1 función RPC, 1 vista y 1 bucket de almacenamiento.

Las llaves se identifican así: PK = Llave Primaria · FK = Llave Foránea · UNIQUE = Restricción de unicidad

Row Level Security (RLS)

Todas las tablas tienen RLS habilitado. Las políticas se evalúan automáticamente en cada query, sin importar de dónde venga la solicitud.

TablaAdminProveedorPúblico
proveedoresALL (SELECT/INSERT/UPDATE/DELETE)SELECT solo su empresa—
personalALLSELECT e INSERT en su empresa; UPDATE solo si estado='rechazado'—
documentos_personalALLALL sobre docs de su personal—
vehiculosALLALL sobre sus vehículos—
grupos_ingresoALLSELECT e INSERT en su empresa—
criterios_evaluacionALLSolo SELECT—
evaluacionesALLSolo SELECT de las suyas—
detalle_evaluacionALLSolo SELECT—
audit_logsSolo SELECT——
notificationsSin restricción especialSELECT y UPDATE solo las suyas—
email_logsALL——
El cliente admin de Supabase (SERVICE_ROLE_KEY) se usa exclusivamente en el backend y bypasea RLS. Se utiliza para operaciones privilegiadas como crear usuarios, enviar notificaciones masivas o hacer updates que el rol de usuario no permitiría.

Endpoints protegidos por token secreto (CRON_SECRET) que se ejecutan de forma programada.

Consulta documentos_personal buscando documentos cuya fecha_vencimiento coincide exactamente con hoy + 30 días o hoy + 60 días. Por cada documento encontrado, envía un correo de alerta al proveedor. Retorna { ok, procesados, enviados }.

GET /api/cron/reporte-mensual

Genera un reporte del mes anterior con: total de registros, conteo por estado (aprobado/pendiente/rechazado), agrupación por empresa, y documentos que vencen ese mes. Lo envía por correo a todos los usuarios con rol='admin'.

Sistema de Auditoría

Todas las acciones críticas quedan registradas en audit_logs con el usuario, la acción, la entidad afectada y metadatos en JSONB.

La función logAudit() en src/lib/audit.ts es llamada desde cada endpoint relevante. Los admins pueden ver el historial completo en /dashboard/auditoria (últimas 200 entradas).

Middleware de Protección de Rutas

El archivo src/middleware.ts intercepta todas las peticiones y gestiona las sesiones.

Ruta / PatrónComportamiento
/login, /auth/*, /registro-publicoPúblicas: pasan sin verificación. Si el usuario YA tiene sesión y va a /login, redirige a /dashboard.
/dashboard/* y todas las demásProtegidas: si no hay sesión activa, redirige a /login.
Archivos estáticos (_next, images, fonts)Excluidos del matcher — no pasan por el middleware.
El middleware refresca la sesión automáticamente en cada request (decodifica el JWT localmente sin hacer llamadas HTTP a Supabase, lo que lo hace muy eficiente).
